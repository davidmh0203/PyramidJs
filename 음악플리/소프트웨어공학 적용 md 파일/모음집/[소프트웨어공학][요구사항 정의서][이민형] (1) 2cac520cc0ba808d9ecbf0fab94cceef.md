# 1. 개요

1.1 문서의 목적

이 문서는 음악 플랫폼간 재생목록을 자동으로 변환하는 웹 서비스 Genie2YT의 요구사항을 정의한다.

이 문서는 개발과정에서의 기능, 비기능, 품질, 인터페이스 요구사항을 명확하게 정의하여 개발자와 이해관계자가 같은 기준으로 시스템을 설계, 구현 할 수 있도록 하는것을 목표로 한다.

1.2 프로젝트 개요

Genie2YT는 사용자가 지니뮤직과 같은 국내 음악 플랫폼의 재생목록을 Youtube Music과 같은 다른 플랫폼으로 자동으로 옮길 수 있도록 하는 웹기반 서비스이다.

Flask 백엔드와 React 프론트엔드를 사용하며, Google OAuth 2.0 인증을 통해  Youtube 계정과 연동한다. 또한  Youtube Data API를 이용해 새로운 재생목록을 생성하고, 재생 목록 리스트에 맞게 자동으로 추가한다. 

1.3 관련 문서, 용어 및 약어

| 구분 | 내용 |
| --- | --- |
| 관련 문서 |  프로젝트 관리계획서, 기능점수 산정서 |
| 용어 | OAuth (사용자 인증 프로토콜), Quota (API 호출 제한 단위), Playlist (재생목록) |
| 약어 | FR (기능적 요구사항), NFR (비기능적 요구사항), API (Application Programming Interface) |

# 2. 요구사항

### 2.1 기능적 요구사항

| ID | 요구사항 | 설명 |
| --- | --- | --- |
| FR-01 | 시스템은 사용자가 Google 계정으로 로그인할 수 있게 해야 한다. | Google OAuth 2.0 인증 과정을 통해 계정 정보를 연동한다. |
| FR-02 | 시스템은 로그인 시 발급된 인증 토큰을 안전하게 저장하도록 한다. | Access Token과 Refresh Token은 암호화된 형태로 관리한다. |
| FR-03 | 시스템은 사용자가 지니뮤직 재생목록의 URL을 입력할 수 있도록 한다. | 입력된 URL을 Flask 서버로 전달하여 데이터를 파싱한다. |
| FR-04 | 시스템은 재생목록에서 곡명과 아티스트 정보를 추출할 수 있어야 한다. | BeautifulSoup을 이용해 HTML 구조를 분석한다. |
| FR-05 | 시스템은 추출된 곡 정보를 JSON 형식으로 변환하도록 한다. | JSON 데이터는 YouTube API 모듈로 전달된다. |
| FR-06 | 시스템은 YouTube API를 이용해 새로운 재생목록을 만들 수 있어야 한다. | `playlists.insert` 요청을 통해 사용자 계정에 재생목록을 생성한다. |
| FR-07 | 시스템은 변환된 곡들을 YouTube 재생목록에 자동으로 추가해야 한다. | 각 곡마다 `playlistItems.insert` API를 호출한다. |
| FR-08 | 시스템은 변환 중 오류가 발생하면 사용자에게 안내 메시지를 보여주어야 한다. | 오류 코드와 원인을 함께 표시한다. |
| FR-09 | 시스템은 변환이 완료되면 성공/실패 수와 처리 시간을 요약해서 보여줘야 한다. | 결과를 시각적으로 표시해 사용자 이해를 돕는다. |
| FR-10 | 시스템은 변환 내역을 데이터베이스에 저장하도록 한다. | 변환 날짜, 성공률, 오류 내용을 기록한다. |
| FR-11 | 시스템은 사용자가 이전 변환 기록을 조회할 수 있도록 해야 한다. | 날짜나 재생목록 이름으로 검색할 수 있다. |
| FR-12 | 시스템은 API 사용 한도를 초과하면 자동으로 재시도해야 한다. | 일정 시간 간격을 두고 최대 3회까지 재시도한다. |
| FR-13 | 시스템은 변환된 재생목록의 링크를 사용자에게 보여주어야 한다. | YouTube 재생목록 URL을 반환한다. |
| FR-14 | 시스템은 변환이 끝나면 완료 알림을 표시해야 한다. | “변환이 완료되었습니다.” 메시지를 출력한다. |
| FR-15 | 시스템은 추후 YouTube 재생목록을 지니뮤직 형식으로 변환할 수 있어야 한다. | 지니뮤직의 “여러 곡 한 번에 찾기” 기능에 사용할 수 있는 텍스트 블록을 생성한다. |

2.2 비기능적 요구사항

| ID | 요구사항 | 설명 |
| --- | --- | --- |
| NFR-01 | 시스템은 50곡 기준으로 10초 안에 변환을 마쳐야 한다. | API 요청과 파싱 시간을 포함한 전체 처리 시간이다. |
| NFR-02 | 시스템은 모든 통신 구간에 HTTPS를 적용해야 한다. | 인증 및 데이터 전송 과정이 암호화되어야 한다. |
| NFR-03 | 시스템은 OAuth 토큰과 API 키를 외부에 노출하지 않아야 한다. | `.env` 파일을 이용해 환경변수로 관리한다. |
| NFR-04 | 시스템은 요청이 실패하면 1초, 2초, 4초 간격으로 최대 3회까지 다시 시도해야 한다. | 서버나 네트워크 오류에 대응하기 위함이다. |
| NFR-05 | 시스템은 오류가 발생하더라도 데이터 손실 없이 복구할 수 있어야 한다. | 변환 로그를 데이터베이스에 저장해 복구 가능하게 한다. |
| NFR-06 | 시스템은 PC와 모바일 브라우저 모두에서 정상 작동해야 한다. | 반응형 웹 화면으로 구성한다. |
| NFR-07 | 시스템은 YouTube API의 일일 사용 한도(10,000 유닛) 내에서 효율적으로 작동해야 한다. | 중복 요청을 줄이고 캐시를 적용한다. |
| NFR-08 | 시스템은 오류가 발생하면 자동으로 로그 파일을 생성해야 한다. | 날짜, 사용자, 오류 내용을 기록한다. |
| NFR-09 | 시스템은 일일 한도를 초과할 경우 사용자에게 알림을 표시해야 한다. | “API 사용 한도 초과” 메시지를 출력한다. |
| NFR-10 | 시스템은 개인정보보호법과 저작권 정책을 지켜야 한다. | 비공개 재생목록이나 타인의 데이터에 접근하지 않는다. |

2.3 인터페이스 요구사항

| ID | 요구사항 | 설명 |
| --- | --- | --- |
| IF-01 | 시스템은 Google OAuth 2.0 인증 서버와 연결되어야 한다. | 클라이언트 ID와 리디렉션 주소를 등록해 인증 과정을 진행한다. |
| IF-02 | 시스템은 YouTube Data API v3와 연동되어야 한다. | 재생목록 생성, 검색, 항목 추가를 위한 API를 사용한다. |
| IF-03 | 시스템은 Flask 서버와 React 프론트엔드가 데이터를 주고받을 수 있어야 한다. | JSON 형식으로 요청과 응답을 처리한다. |
| IF-04 | 시스템은 데이터베이스와 연결되어야 한다. | SQLite를 기본으로 사용하고, 필요 시 PostgreSQL로 확장한다. |
| IF-05 | 시스템은 지니뮤직 웹 페이지에서 곡 정보를 가져올 수 있어야 한다. | BeautifulSoup를 사용한다. |

# 5. 기타 고려사항

- Google API 정책 변경 시 시스템이 영향을 받을 수 있다.
- YouTube API 사용 한도 초과 시 쿼터 증설이나 대체 API 사용이 필요하다.
- 지니뮤직 웹 구조가 변경되면 파싱 로직을 수정해야 한다.
- 사용자 정보와 로그 데이터는 민감 정보가 포함되지 않도록 주의한다.
