# **목차**

# 1. 개요(Introduction)

## 1.1 문서의 목적

본 문서는 음악 스트리밍 플랫폼 간 재생목록을 이전하는 서비스 ‘TuneMyMusic’을 분석하고,

해당 서비스를 모델로 한 유사 프로젝트 개발 계획을 수립하기 위한 프로젝트 관리계획서이다.

이 문서는 소프트웨어공학 수업의 과제 목적에 따라 작성되었으며, 서비스 구조, 개발 절차, API 활용 전략, 품질 및 리스크 관리 계획을 포함한다.

## 1.2 프로젝트 개요

TuneMyMusic은 Spotify, Apple Music, YouTube Music, Deezer, Tidal 등 다양한 음악 플랫폼 간 재생목록(Playlist)을 손쉽게 옮길 수 있는 웹 기반 서비스이다.사용자는 로그인만으로 자신의 음악 데이터를 선택하고, 대상 플랫폼으로 자동 전송할 수 있다.추가로, “Sync 기능”을 통해 두 서비스 간 재생목록을 자동 동기화(주기적 업데이트)할 수 있다

## 1.3 관련 문서, 용어 및 약어

- API (Application Programming Interface) : 각 음악 플랫폼의 데이터 접근 인터페이스
- OAuth 2.0 : 사용자 계정 인증 방식 (Spotify, YouTube 등 공통 사용)
- Quota : YouTube Data API 등에서 요청 제한을 나타내는 단위

- 기본 할당량: 10,000단위 / 하루

- 효율적 사용 전략
    - 동일 곡 중복 요청 방지 (Cache 적용)
    - 사용자 로그인별 API 키 분리 → Quota 분산
    - Google Cloud Console에 증설 요청서 제출
    - 실패 시 재시도(log 기록 기반)

# 3. 조직 구성(Organization)

## 3.1 팀 구조

### **팀 구조 선정 전략**

### 고려 요인

- 프로젝트는 다수의 외부 API(YouTube, Spotify, Apple Music 등)와 연동해야 함
- 백엔드-프론트엔드 간 협업이 필수이며, 일정 관리와 커뮤니케이션이 중요함
- 실시간 회의와 문서 기반 협업이 병행되는 형태가 효율적임

### 선택 가능한 팀 구조 유형

| 팀 구조 유형 | 특징 | 장점 | 단점 |
| --- | --- | --- | --- |
| **중앙집중형** | PM 중심 의사결정, 수직적 보고 체계 | 명확한 통제, 일정 관리 용이 | 유연성 부족, 개발 속도 저하 가능 |
| **분산형(탈중앙형)** | 각 파트 자율 진행, 문서로 공유 | 유연하고 신속한 대응 가능 | 조율 및 통합 어려움 |
| **하이브리드형** | 핵심 결정은 중앙집중식, 개발은 분산식 | 통제와 유연성의 균형 | 관리 복잡성 증가 가능 |

**최종 선정 구조: 하이브리드 팀 구조 (Hybrid Team Structure)** 

### 선정 이유

- TuneMyMusic 유사 시스템은 플랫폼 간 복수 API 연동이 핵심이므로, 각 플랫폼별 분할 개발(분산형 특성) 과 통합 검증 및 일정 관리(중앙집중형 특성)이 요구된다.
- 백엔드(API) / 프론트엔드(UI) / 품질관리 / 지원그룹이 서로 병행되지만,최종 통합 및 품질 승인은 PM → PL → TL → QE 단계를 거치는 중앙 방식을 유지한다.
- 따라서 하이브리드형 팀 구조가 기술적 효율성과 관리적 통제의 균형을 가장 잘 충족한다.

# 8.기타고려사항

### **OAuth2 인증 및 보안 관리**

- 사용자 계정 접근 시 OAuth 인증 과정에서 Access Token 만료 / 갱신 실패가 발생할 수 있음.
- 보안상 Token, API Key는 DB에 직접 저장하지 않고 환경 변수(.env) 또는 암호화 스토리지로 관리.
- 토큰 자동 갱신 로직 구현 및 토큰 만료 시 사용자 재로그인 유도
- HTTPS 기반 통신 필수 (인증 정보 암호화 전송).

### **외부 API 정책 변경 리스크**

- YouTube, Spotify, Apple Music 등 외부 API 의존도가 높아 정책 변경 시 서비스 장애 가능.
- Google의 YouTube Data API는 하루 할당량(10,000단위) 제한이 존재하며, 초과 시 API 요청 차단 발생.
- 대응 방안: API 호출 최적화 / 캐싱 / 사용자별 인증키 분리 / Quota 증설 요청 자동화.

### **데이터 매칭 정확도**

- 서로 다른 플랫폼 간 트랙 이름/아티스트 표기법 차이로 매칭 실패 가능.
- 문자열 유사도(Levenshtein Distance, Jaro-Winkler) 기반 검색 보정 알고리즘 적용 필요.
- 예: “아이유(IU)” vs “IU” / “사랑은 아파도 (Love Hurts)” 같은 변형 케이스 대응.
- 서버 부하 최소화를 위해 매칭 결과는 캐시에 저장하며, 동일 요청 시 API 호출 없이 재사용한다.

### **저작권 및 데이터 접근 제한**

- 음악 데이터(재생목록, 곡명, 앨범아트)는 각 플랫폼의 저작권 보호 대상.
- TuneMyMusic처럼 “사용자 계정 내 플레이리스트 복제”는 허용되지만, 곡 파일 다운로드나 외부 공유는 저작권 위반 소지 있음.
- 대응 방안: “메타데이터(제목, 아티스트명, ID)”만 이동하고, 음원은 대상 플랫폼 내에서만 재생되도록 제한.
